Why

Why are we building this? What problems are we trying to solve?

As a working photographer I ran into the same friction after every shoot. Clients needed to review proof images and choose which ones they wanted retouched, but the available tools were either too expensive or broken for how real selection decisions actually get made.

The SaaS proofing platforms on the market charged $200–300/year with storage caps, gallery expiration dates, and a one-at-a-time viewing experience that hadn't evolved in years. The core design failure: when a client is choosing between subtle lighting variations or similar poses, they need to see those images together — not flip between them trying to hold one in memory while viewing another.

User interviews confirmed what I was experiencing firsthand. Clients weren't struggling with the concept of rating images — they were struggling with the inability to hold multiple similar images in view at the same time. The market had built rating tools. Nobody had built a real comparison tool designed for the end client.

I reviewed Adobe Lightroom's Survey Mode and Capture One's online collaboration gallery as reference points. Both had relevant pieces but neither solved the client-facing problem: Survey Mode is desktop-only and client-inaccessible; Capture One limits comparison to two images.

Who, Where, When

Primary User

The photographer's clients — people who hired a photographer for portraits, events, or commercial work and need to select which images they want retouched. They have no photography background and varying technical comfort. The tool must work fluently for the least technical person in the room, on whatever device they pick up. No tutorial. No account. Just a link that works.

Secondary User

The photographer. Every minute a client spends uncertain or sending back-and-forth emails is time spent managing communication instead of editing. The tool needs to get clients to a confident, exportable selection without hand-holding.

Where & When

Hosted on the photographer's own website. Clients receive a link via email immediately after a shoot, when proof images are delivered, and review 100+ photos in a single session at their own pace — on desktop or mobile, with no friction before they can start.

Process — AI-Assisted Design & Build

How Claude AI changed the workflow

This project didn't follow a traditional UX process. Rather than designing flows in Figma, writing interaction specs, and handing off to a developer, I used Claude AI for vibe coding — describing design intent in plain language and iterating on working browser prototypes in real time. It's how I now operate as a designer who can also build: faster feedback loops, complex state problems caught before they're expensive, and a 2-day path from concept to shipped production tool.

The core advantage over a Figma handoff

The traditional pipeline creates a gap between design intent and built reality — especially for nonlinear, stateful interfaces. You can design a happy path beautifully and still not hit interaction failures until a developer has already built it. With Claude, I was testing real state interactions from the very first iteration, not static mockups. The feedback loop collapsed from days to minutes.

State problems caught during live prototyping

This interface has three interconnected views — the main gallery grid, the single-image lightbox, and compare mode — all sharing the same state. These problems only surface when testing live interactions, not reviewing a Figma file:

Selection + rating sync across all views

Rating an image in the compare lightbox needed to update simultaneously in the main grid, the lightbox header, and the active filter view — without a page refresh. Invisible in a Figma prototype. Immediately testable in a live Claude prototype.

Filter reactivity to live rating changes

The star filter had to react to live rating changes — downrate an image below the threshold and it disappears from the filtered view instantly, even inside the lightbox or compare mode. One iteration to discover and resolve because we tested real state transitions.

Zoom & pan state per-image in compare mode

Each of up to 5 compare panels needed its own independent zoom and pan state — zooming one panel couldn't affect the others. This isolation requirement only became clear once tested live in the browser.

Traditional Figma → Handoff

  • Design happy-path flows in Figma
  • Write interaction specs for stateful edge cases
  • Hand off to developer
  • Developer builds v1
  • QA reveals cross-view state bugs
  • Revise specs, schedule fixes
  • Multi-week cycle to shipped product

How

How can we measure impact and growth to validate the solution?

Client feedback was immediate and consistent: the comparison feature was the thing people talked about. Several described the selection process as finally feeling manageable. Nobody needed instructions — they opened the link and started rating.

When I shared the tool with other photographers in my community, multiple photographers said multi-image compare was something they had been wanting for years. One noted their client turnaround time for selections dropped noticeably — fewer back-and-forth emails, faster decisions, smoother handoffs to retouching.

What started as a personal workflow solution has since generated unsolicited purchase interest from photographers in the community — a signal that the comparison gap it fills is wider than one studio. Future plans include a stable distributable release, optional client commenting, and a streamlined setup experience for non-developer photographers.
